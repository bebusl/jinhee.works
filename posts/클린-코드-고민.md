## 핵심 한 줄 요약

> 재사용이 목적이 아닌 분리는 → 레이어를 낮추지 말고, “해당 feature 내부에서” 구조화한다.

즉,

- ❌ 재사용 안 되는데 widgets / shared / entities로 뺀다 → FSD 위반에 가까움
- ✅ feature 하나가 너무 커서 이해 안 될 때 → **feature 내부에서만 쪼갠다**

## 왜 FSD에서 “함부로 쪼개지 말라”고 하냐면


FSD의 기본 철학은 이거야:

> “구조적 분리는 책임 경계를 기준으로 하고,
>
> **가독성 문제**는 레이어가 아니라 **폴더 내부에서 해결**하라”
>
>

그래서 문서에서 말하는


“재활용 안 되면 feature/widget로 나누지 마라”는 말은 사실:

> ❌ 레이어 경계를 만들지 마라
>
> ✅ **파일/폴더는 마음껏 나눠도 된다**
>
>

라는 뜻에 더 가까워.


---


## 거대한 폼의 정석적인 FSD 구조


### ❌ 안 좋은 예 (의미 없는 레이어 분리)


```plain text
widgets/
  BigForm/
features/
  FormSectionA/
  FormSectionB/
```


→ 재사용도 안 되는데 “폼이 커서”라는 이유로 레이어 쪼갠 경우


---


### ✅ FSD에서 권장되는 방식


```plain text
features/
  create-product-form/
    ui/
      CreateProductForm.tsx
      sections/
        BasicInfoSection.tsx
        PriceSection.tsx
        ImagesSection.tsx
      fields/
        NameField.tsx
        PriceInput.tsx
    model/
      useCreateProductForm.ts
      validation.ts
      types.ts
```


👉 **경계는 하나 (create-product-form)**


👉 내부는 **가독성과 복잡도 기준으로만 분해**


---


## 판단 기준을 이렇게 두면 좋아


### 1️⃣ “이게 다른 feature에서도 의미가 있나?”

- YES → `shared / entities / widgets` 고려 ✅
- NO → **feature 내부에 둔다** ✅

### 2️⃣ “비즈니스 개념 단위인가?”

- 가격 정책, 회원 상태, 주문 같은 개념 → `entity`
- 단순히 이 폼을 구성하는 덩어리 → `feature 내부`

### 3️⃣ “이 분리가 의존성 방향을 바꾸는가?”

- 바뀐다 → 안 나눔 ❌
- 내부 구현만 쪼갬 → 문제 없음 ✅

---


## “feature 단에서도 또 나누는 게 이상하지 않나?” → 전혀 아님


오히려 실전 FSD에서는 이런 구조가 흔해:


```plain text
feature/
 ├─ ui/
 │   ├─ index.tsx
 │   ├─ sections/
 │   └─ fields/
 ├─ model/
 ├─ lib/
 └─ config/
```


FSD는 **“레이어 수를 줄이는 아키텍처”지
“파일 수를 줄이는 아키텍처”가 아냐**.


---

>

## pages에서 해결하던 걸 FSD로 옮길 때의 정답


### ✅ 제일 중요한 포인트

> “pages에 있던 복잡도”는  
> 그대로 pages → feature 안으로 옮기면 된다.

즉,

- pages는 **조합만**
- 복잡한 폼 로직 / UI 분해는 **feature가 담당**

---


## Before → After 감각 맞추기


### ❌ pages에 다 들어있는 상태 (익숙한 상태)


```plain text
pages/
 └─ createProduct/
    └─ page.tsx   // 폼 UI, 상태, validation 전부 여기
```


이 구조 자체가 “잘못”은 아니었음.


단지 **규모가 커져서 관리가 힘들어진 것**뿐.


---


### ✅ FSD로 옮긴 뒤의 이상적인 상태


```plain text
pages/
 └─ create-product/
    └─ page.tsx   // layout + feature를 가져와서 조합

features/
 └─ create-product-form/
    ├─ ui/
    │   ├─ CreateProductForm.tsx
    │   └─ sections/
    │       ├─ BasicInfoSection.tsx
    │       ├─ PriceSection.tsx
    │       └─ ImageSection.tsx
    ├─ model/
    │   ├─ useCreateProductForm.ts
    │   ├─ validation.ts
    │   └─ types.ts
    └─ index.ts
```


👉 **pages → composition 역할만**


👉 **전에 pages에서 하던 모든 더러운 일은 feature로 이동**


이게 정확히 FSD가 의도한 사용법이야.


---


## 그렇다고 “pages에서 feature 여러 개 쓰면 안 된다” ❌


많이들 오해하는 포인트인데:

- **한 page = feature 하나** ❌ 아님
- 한 page는 보통:
    - 여러 feature
    - 필요하면 widgets
    - layout

을 **그냥 조합하는 곳**이야.

## 왜 “재활용”만 feature 기준으로 보면 안 되냐면


FSD에서 말하는 feature의 본질은:

> 사용자가 의도를 가지고 실행하는 행동

이게 **다른 곳에서 다시 쓰이든 / 한 번만 쓰이든**은

### 즉 feature의 진짜 기준


| 기준           | 로그인      | 회원가입    |

| validation   | 간단       | 복잡      |
| pages에 두기 적합 | ✅        | ❌       |
| feature로 분리  | optional | ✅ 강력 추천 |

**재사용 여부랑은 상관 없음**.

## 헷갈릴 때 쓰는 한 방 기준


이 문장으로 설명 가능하면 ui에 둬도 된다 👇

> “이건 사용자에게 보여주기 위한 처리다”

안 되면 model이다 👇

> “이건 업무 규칙에 대한 판단이다”

**이건 한 번 꼭 정리하고 넘어갔음 좋겠음.** 

> 2026.2.1 업데이트

ui — everything related to UI display: UI components, date formatters, styles, etc.
api — backend interactions: request functions, data types, mappers, etc.
model — the data model: schemas, interfaces, stores, and business logic.
lib — library code that other modules on this slice need.

## 전체 한 줄 요약

- **ui**: React + 화면에 강하게 묶임
- **api**: 서버와의 통신 규약
- **model**: 도메인 규칙과 상태 (UI·서버 모두로부터 최대한 독립)
- **lib**: 범용 도구
- **config**: 환경/플래그

---


## 1. `ui` — “React에 묶여 있으면 여기”

> 기준: React / 렌더링 컨텍스트가 없으면 의미가 없는가?

### 들어가도 되는 것

- 컴포넌트
- UI 상태 관리 훅 (`useState`, `useReducer`, `useForm` 등)
- **서버 데이터를 ‘화면 관점’으로 쓰는 훅**

```typescript
// ui/useProductList.ts
export function useProductList() {
  const query = useQuery({
    queryKey: ['products'],
    queryFn: fetchProducts,
  })

  return {
    products: query.data,
    isLoading: query.isLoading,
    isEmpty: query.data?.length === 0,
  }
}
```


👉 `useQuery` 써도 **ui에 둬도 됨**


왜냐면:

- React 훅이고
- loading / empty / error 같은 **화면 상태를 만들고 있음**

### 들어가면 안 되는 것

- fetch 함수 자체
- DTO 정의
- 도메인 규칙 계산

---


## 2. `api` — “백엔드 계약서”

> 기준: 서버 스펙이 바뀌면 같이 바뀌는가?

### 여기에 들어가야 하는 것

- HTTP 호출 함수
- Request / Response DTO
- 서버 전용 타입
- 서버 기준 데이터 구조

```typescript
// api/product.api.ts
export async function fetchProducts(): Promise<ProductResponseDTO[]> {
  return http.get('/products')
}

export interface ProductResponseDTO {
  id: number
  price: number
  created_at: string
}
```


### mapper는 어디?


**결론부터 말하면:** _**보통 api 쪽이 맞음**_


### 이유

- DTO → FE 모델 변환은 **서버 계약을 해석하는 작업**
- 서버 필드명 변경되면 mapper가 깨짐 → api 책임

```typescript
// api/product.mapper.ts
export function mapProduct(dto: ProductResponseDTO): Product {
  return {
    id: dto.id,
    price: dto.price,
    createdAt: new Date(dto.created_at),
  }
}
```


👉 이 mapper를 `model`에 두면


**model이 서버 DTO를 알게 되는 순간 의존성 방향이 깨짐**


---


## 3. `model` — “UI도 서버도 모르게”

> 기준: React도 모르고, HTTP도 모르게 유지할 수 있는가?

### 여기에 들어가는 것

- FE에서 사용하는 도메인 타입
- 비즈니스 규칙
- 계산 로직
- 상태 머신 / store (zustand, jotai 등)

```typescript
// model/product.ts
export interface Product {
  id: number
  price: number
  createdAt: Date
}

export function isDiscountable(product: Product) {
  return product.price > 10000
}
```


### ❌ api 호출은?


**model에서 직접 호출하면 안 됨**


왜냐면:

- model이 외부 세계(서버)에 의존
- 순수성 깨짐
- 테스트 난이도 급상승

👉 “비즈니스 로직인데 api가 필요함” →


대부분 **그건 진짜 도메인이 아니라 유스케이스 레벨**


이 경우는 보통:

- `service`
- `feature`의 `model`
- 혹은 `application` 레이어가 따로 있음 (FSD 확장)

---


## 4. `lib` — “맥락 없는 도구”

> 기준: 이 프로젝트/도메인 몰라도 쓸 수 있는가?
- 날짜 포맷 유틸
- debounce / throttle
- safeParse
- invariant

```typescript
// lib/formatDate.ts
export function formatDate(date: Date, format = 'YYYY-MM-DD') {}
```


❌ 여기에 두면 안 되는 것

- product 관련 유틸
- price 계산

    → 그건 model임

## 5. `config` — 네 말이 거의 맞음

> 기준: 행동이 아니라 설정인가?
- feature flag
- env mapping
- 상수 설정

```typescript
export const ENABLE_NEW_CHECKOUT = true
```


실무에선 생각보다 자주 쓰이긴 함

## 네가 헷갈린 포인트 정리


### Q1. `useQuery` 훅 → ui vs api?

- **훅 자체**: `ui`
- **queryFn (fetch)**: `api`

### Q2. DTO → FE 모델 mapper → api vs model?

- **대부분 api**
- model은 “서버에서 왔는지조차 모르게” 유지하는 게 목표

### Q3. model에 api 호출이 필요한 경우?

- 구조가 잘못됐을 가능성 큼
- 그 로직은 보통 **유스케이스 레이어**

## 판단용 체크리스트 (이거 기억해)

> 파일 하나 만들기 전, 이 질문만 해도 80% 해결됨
1. React 없이 의미 있나? → **ui**
2. 서버 스펙 바뀌면 같이 깨지나? → **api**
3. UI/서버 둘 다 몰라도 되나? → **model**
4. 어디서든 써도 되나? → **lib**
5. 값만 있고 로직 없나? → **config**

지금 수준에서 이 정도로 경계 고민하는 건 정상이고,


오히려 안 헷갈리면 그게 더 위험한 단계야.
