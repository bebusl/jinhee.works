## **[PHASE1. 회사 적응기 ]**


일단 서비스 자체에 대한 이해가 필요했다. 나는 평상시에 숏폼, 릴스, 라이브 등을 통해 물건을 잘 구매하지 않았었는데 우리 회사 도메인 자체가 라이브 커머스였기 때문에 해당 도메인에 대해 좀 익숙해져야 했다. N사의 라이브 쇼핑이나 유튜브 숏폼을 많이 참고했었던 것 같다.[이 때 소비내역보면 평상시보다 소비 내역이 꽤 많아ㅎㅎ]


기술적으로는 일단 나에게 회사 서비스는 꽤 커다란 코드 베이스였고, 정리된 히스토리가 없었기 때문에 코드간 관계, 그리고 서비스간 관계를 파악하는게 최우선이었다. 기능 정의가 최신화되지 않은 상황에서 최선의 ? 개발 환경에서 온갖 기능을 다 켜서 파악하고 노션에 정리하는 작업을 했다. [이 작업이 추후 다른 사람들에게 온보딩을 할 때 도움이 됐다, 물론 다른 사람에게 설명할 때도 매우 도움 됐음]


업무 관련해서는 일단 결재 플로우가 어떻게 되는지 파악하려고 했었던 것 같다. 


질문한 내용은 죄다 노션에 정리했었다.


의문이 드는 점이 있으면 무조건 정리하려고 했었던 것 같다.

> Level1. 질문하고, 정리하는 법 터득. 회사 프로세스가 어떻게 돌아가는지 파악하게 됐다!

# 2. 회사생활하면서 어떤 생각을 했었나요?


회사에서 일하며 느낀 가장 큰 변화는, ‘필요한 개발자 상’에 대한 인식이었다.

- 기술 실력만으로는 문제를 해결할 수 없다
- 질문을 잘 정리하는 사람이 신뢰를 얻는다
- 혼자 완벽하게 이해하려 하기보다, 공유하는 편이 빠르다
- 설명 능력은 개발자의 중요한 역량 중 하나다

나는 종종 스스로를 과소평가하며 흔들렸지만,


적어도 **성실하게 이해하려는 태도**만큼은 유지하려고 했다.


# 3. 왜 회사를 퇴사하게 됐나요?


퇴사의 직접적인 이유는 건강 문제였다.


장기간 치료가 필요했고, 업무와 병행하기에는 현실적인 한계가 있었다.


아쉬움은 있었지만, 몸을 먼저 정비하는 것이 더 중요하다고 판단해 퇴사를 결정했다.


또 한편으로는, 다른 환경에 대한 호기심도 분명히 존재했다.

## 들어가며


2023년 7월, 나는 소규모 스타트업에 프론트엔드 개발자로 입사했다.


인턴과 교육 경험은 있었지만, 정식으로 회사에 소속되어 일하는 것은 처음이었다. ‘이제 진짜 사회인이 되는구나’라는 생각에 약간의 긴장과 기대가 섞여 있던 시기였다.


이 글은 첫 회사에서 약 2년간 일하며 어떤 역할을 맡았고, 어떤 방식으로 성장했고, 무엇을 배우게 되었는지를 정리한 기록이다. 기억이 더 흐려지기 전에 남겨두고 싶었다.

## 1. 무엇을 했나요?


내 회사 생활은 크게 세 단계로 나눌 수 있다.

- 회사 적응기
- 백오피스 담당기
- 백오피스를 넘어 여러 서비스를 맡게 된 시기

각 단계는 단순히 맡은 일이 달라졌다기보다, 일하는 방식과 관점이 바뀌는 전환점이었다.


---


### PHASE 1. 회사 적응기


### 문제

- 라이브 커머스 도메인에 대한 이해 부족
- 히스토리가 정리되지 않은 비교적 큰 코드베이스
- 최신화되지 않은 기능 정의 문서

나는 평소 숏폼이나 라이브 커머스를 통해 물건을 구매하는 편이 아니었기 때문에, 서비스 도메인 자체가 낯설었다. 먼저 사용자 관점에서 도메인을 이해하기 위해 타사의 라이브 쇼핑, 숏폼 콘텐츠를 의식적으로 많이 참고했다.


기술적으로는 코드 구조와 서비스 간 관계를 파악하는 것이 가장 시급했다. 기능 정의가 정리되어 있지 않은 상황이었기 때문에, 실제로 가능한 기능을 하나씩 직접 켜 보고 흐름을 추적했다.


### 내가 한 일

- 기능을 직접 실행하며 서비스 플로우 파악
- 코드 간 관계, 서비스 구조를 노션에 정리
- 업무 중 나온 질문과 답변을 모두 기록
- 결재 및 협업 프로세스 정리

이 과정에서 ‘질문 → 정리’를 반복했다. 이해되지 않는 상태로 넘어가는 것을 최대한 피하려고 했다.


### 결과

- 서비스 전반 구조에 대한 이해 확보
- 이후 신규 인원 온보딩 시 참고 자료로 활용
- 개인적으로도 설명 능력이 눈에 띄게 개선됨
> Level 1. 질문하고 정리하는 법을 배우며, 회사가 어떻게 돌아가는지 이해하기 시작했다.

---


### PHASE 2. 백오피스 담당기


회사 업무에 어느 정도 익숙해진 뒤, 서비스 백오피스 프론트엔드 개발을 전담하게 되었다.


### 문제

- 옵션과 분기가 많은 복잡한 UI
- 비슷한 로직이 반복되는 코드 구조
- 기능 추가 시 수정 범위가 넓어지는 구조

### 내가 한 일

- 기능 단위로 책임을 나누는 리팩토링 지속
- 분기 로직을 단순화하고 중복 제거
- 빌링 관련 기능, 리니어 관련 기능 개발
- 내부 사용성을 개선하기 위한 디버깅 도구 개발

리팩토링을 진행하면서 ‘코드를 줄이는 것’보다, **이 코드가 어떤 책임을 가져야 하는지**를 더 많이 고민하게 되었다.


### 결과

- 코드 변경 시 영향 범위 예측이 쉬워짐
- 스스로의 코드에 대한 한계를 명확히 인식
- 실력에 대한 과도한 자신감 대신, 기준을 세우는 법을 배움
> Level 2. 리팩토링은 기술 문제가 아니라 사고방식의 문제라는 걸 체감했다.

---


### PHASE 3. 이것저것 담당기


이후 백오피스뿐 아니라, 외부에 제공되는 `View`라는 서비스도 함께 맡게 되었다.


이 서비스는 백오피스에서 설정한 위젯 구성을 기반으로, 외부 사이트에 삽입되는 웹 SDK 형태의 서비스였다.


### 문제

- 다양한 외부 환경에 대응해야 하는 구조
- 위젯 및 페이지 수 증가
- 몇 px 차이로도 새로운 변형이 필요한 상황

### 내가 한 일

- 위젯과 페이지 구조를 재정비
- 공통 요소와 커스터마이징 포인트 분리
- 비개발 직군에게도 이해 가능한 설명 방식 고민

이 시기부터는 ‘내가 이해하는 코드’보다 ‘다른 사람이 이해할 수 있는 코드’를 더 의식하게 되었다.


### 결과

- 협업 시 커뮤니케이션 비용 감소
- 코드 구조를 설명하는 능력 향상
- 개발 외 이해관계자를 고려한 설계 경험
> Level 3. 협업 가능한 코드 구조와 설명 능력의 중요성을 체감했다.
